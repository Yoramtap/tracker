## Quarter-by-Quarter Plan (Yoram-Focused)

## Q1: Foundation And Baseline (What You Need To Achieve First)
Primary outcome: establish control systems so the rest of the year is executable.

What you need to achieve:
1. Publish and run structured rituals (weekly + sprint + monthly) with clear attendance expectations.
2. Ratify code standards and linting policy; convert to adoption checklist per repo/team.
3. Stand up SFD-to-release baseline report and publish it on a fixed cadence.
4. Implement reduced workflow for non-high-risk releases and run at least one pilot.
5. Define Broadcast long-term bug criteria and start a dedicated tracking board.

Definition of done for Q1:
1. Ritual attendance rate >= 85% and meeting cadence is stable for at least 2 sprints.
2. 100% target repos have standards/linting documented and enabled.
3. Baseline release-time report is published at least biweekly.
4. Non-high-risk release workflow is used by at least one release train.
5. Broadcast long-term bug board is active with owner + SLA rules.

At-risk signals to watch:
1. Rituals happen but no actions are closed sprint-over-sprint.
2. Standards are approved but repo adoption stalls.
3. Baseline metrics are missing timestamps or are not trusted by teams.

## Q2: Adoption And Process Discipline
Primary outcome: move from setup to repeatable execution.

What you need to achieve:
1. Product teams define release-stage goals (Q2) and lock Q4 improvement targets.
2. Implement product processes for Shift by end of Q2.
3. Complete Q2 testing milestones (Titanium and Orchestration).
4. Start formal Broadcast architecture/workstream checkpoints.

Definition of done for Q2:
1. Stage-goal scorecard is reviewed every sprint.
2. Shift process is in use across planned squads.
3. Coverage trendline is rising and evidence is auditable by team.
4. Broadcast workstream checkpoint cadence is active and documented.

## Q3: Scale And Stabilize
Primary outcome: convert practices into measurable backlog and velocity improvements.

What you need to achieve:
1. Frontend (new UI) reaches its testing milestone.
2. Broadcast backlog/aging reduction plan shows sustained improvement.
3. Monthly velocity review produces corrective actions with owners.

Definition of done for Q3:
1. Highest+high trend is stable or declining sprint-over-sprint.
2. Broadcast long-term bugs decrease quarter-over-quarter.
3. Velocity variance narrows compared with Q1/Q2 baseline.

## Q4: Close And Harden
Primary outcome: hit year-end targets and lock behavior for 2027.

What you need to achieve:
1. Broadcast testing milestone reaches target by Q4.
2. Broadcast architecture/refactor changes are completed and reviewed.
3. Zero long-term bug target for Broadcast is achieved (or within pre-approved tolerance).
4. 2027 carry-forward operating model is documented and approved.

Definition of done for Q4:
1. Long-term Broadcast bugs are at/near zero.
2. Release lead-time metrics are better than Q1 baseline.
3. Rituals, reporting, and ownership model are retained for next year.

## Operating Cadence
- Weekly:
1. Quality ritual checkpoint (attendance + blockers).
2. Broadcast bug/aging review.

- Sprint (every 2 weeks):
1. KPI delta review.
2. Risk register review + mitigations.
3. Next sprint commitments lock.

- Monthly:
1. Executive summary for sponsors (Dan, Koen).
2. KPI health: on-track / at-risk / off-track.

## Risks And Mitigations
1. Risk: Standards are ratified but not adopted.
   Mitigation: team-level adoption checklist + monthly audit.
2. Risk: Coverage targets drive low-value tests.
   Mitigation: include critical-path test quality criteria, not only percentages.
3. Risk: Broadcast queue remains blocked by UAT.
   Mitigation: dedicated unblock lane + aging caps + weekly owner accountability.
4. Risk: Release goals conflict with delivery pressure.
   Mitigation: non-high-risk fast lane with clear entry criteria.
